## v0.6.0 — Hardened Enforcement

**Goal:** Make AxiomGuard enterprise-audit-ready with deterministic error mapping,
extraction transparency, and simple date-bound constraints.

| Feature | Description | Priority |
|---------|-------------|:--------:|
| **Deterministic error mapping** | Z3 UNSAT returns hardcoded `message` from YAML — zero LLM translation of errors | Critical |
| **Extraction transparency** | Always expose raw extracted claims before Z3 processes them — full audit trail | Critical |
| **Date-bound constraints** | `valid_from` / `valid_until` on any rule — temporal norm tracking so rules stay current as norms change | High |
| **Negation rule type** | `type: negation` with `must_not_include` — human-readable prohibition rules | High |
| **Extraction bias audit** | Deterministic keyword check after Layer 1 — flag when protected attributes (gender, race, etc.) appear in extracted claims. No LLM, just pattern matching. Ref: Resnik (2025) | High |
| **Claim classification metadata** | Relations declared as `definitional`, `contingent`, or `normative_risk` in YAML — helps downstream audit distinguish fact types that LLMs cannot separate (Resnik 2025) | Medium |
| **Enhanced normalization** | Strip titles/suffixes/articles deterministically — "Dr. Smith" → "smith" | Medium |
| **Confidence scoring** | `Claim.confidence` field — route low-confidence extractions to human review before Z3, separating "LLM was uncertain" from "LLM was wrong" | Medium |
| **Multi-platform docs** | Windows + macOS + Linux install instructions | Medium |

**Key principle:** Every error message a user sees was written by a human domain expert,
not generated by an LLM. Z3 returns the `message` field verbatim.

## v0.7.0 — Advanced Rules, Tournament Mode & Axiom Studio

**Goal:** Cover complex enterprise constraints, solve the blank-page problem with
Tournament-style rule derivation, and provide a visual rule authoring UI.

| Feature | Description | Priority |
|---------|-------------|:--------:|
| **Tournament mode** | Multi-strategy candidate generation → Z3 conflict detection → human arbitration → approved `.axiom.yml`. Solves the blank-page problem of BYOR. See [research doc](research/v070_tournament_rule_derivation.md) | Critical |
| **Comparison rules** | `loan_amount <= salary * 5` — cross-relation arithmetic | High |
| **Cardinality constraints** | `at_most: 2` / `at_least: 1` — bounded counting | High |
| **Rule composition** | `all_of` / `any_of` / `none_of` combinators in YAML | High |
| **Axiom Studio** | Streamlit UI: Manual Editor + Tournament Mode + Rule Tester | High |
| **LangChain integration** | `AxiomGuardChain` — drop-in for LangChain RAG pipelines | Medium |
| **LlamaIndex integration** | `AxiomGuardPostprocessor` — same priority as LangChain | Medium |
| **REST API wrapper** | FastAPI server for language-agnostic verification | Medium |

**Tournament mode is NOT auto-rule-generation.** The LLM generates *candidates*
(untrusted), Z3 detects *conflicts* (mathematical proof), and humans make
*normative decisions* (authority). Every approved rule has an audit trail:
source excerpt → strategy → conflict resolution → human sign-off.

**Axiom Studio is a rule EDITOR with Tournament support.**
Tab 1: Manual YAML editor with live preview and testing.
Tab 2: Tournament Mode — upload document, generate candidates, resolve conflicts, export.

**Academic foundation:** Resnik (2025) argues LLMs cannot distinguish definitional,
contingent, and normatively unacceptable patterns. Tournament mode addresses this by
extracting from multiple angles, surfacing conflicts via Z3, and letting humans
make the normative calls that LLMs structurally cannot.

## What We Deliberately Do NOT Build

| Feature | Why Not |
|---------|---------|
| Pre-packaged industry rulesets (HIPAA, OWASP, etc.) | Legal liability — we provide the engine, clients provide the rules |
| **Unsupervised** auto-rule generation from documents | LLMs hallucinate constraints — creates fake citations and liability. Tournament mode generates *candidates*, but every rule requires explicit human approval |
| LLM-powered error explanations | Hallucinated debugging — Z3 returns hardcoded `message` only |
| LLM-based bias detection | Resnik (2025): bias is structural in LLMs — using LLM to detect its own bias is circular. We use deterministic keyword audit + human-authored normative rules instead |
| Linear Temporal Logic (LTL) | Over-engineered — simple `valid_from`/`valid_until` covers 95% of needs |
| Embedding-based entity resolution | Breaks deterministic guarantee — alias dict only |
